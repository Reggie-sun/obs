有用，但不是“全局控制变量”，而是“阶段性控制变量”。

你担心的是对的：后面的改动当然会影响前面的结果。chunk 一改，embedding/index/citation 都会变；router 和 rerank 也会改变最终指标。所以这套顺序的目的，不是保证“后面永远不影响前面”，而是保证我们在每一刀下去时，知道变化主要是从哪一层来的。

仓库里之所以定成这个顺序，是因为每一层改动的“破坏半径”不同：

1. 先做 baseline / supplemental，因为这是最轻的一层，主要在在线判定和触发逻辑上收口，不必先动入库底座。RETRIEVAL_OPTIMIZATION_PLAN.md (line 34) RETRIEVAL_OPTIMIZATION_PLAN.md (line 127)
2. chunk 放第二层，因为它会影响 parsed/chunks/embedding/index，一旦先改它，前面所有召回问题都会和切块问题缠在一起，很难判断根因。RETRIEVAL_OPTIMIZATION_PLAN.md (line 167)
3. router / hybrid 放第三层，因为它是在“候选已经相对干净”的前提下调分流和权重；如果候选本身脏，这层结论不稳。RETRIEVAL_OPTIMIZATION_PLAN.md (line 223)
4. rerank 最后做，因为它只是在已有候选上做最后排序，过早介入最容易掩盖上游问题。RETRIEVAL_OPTIMIZATION_PLAN.md (line 242) AGENTS.md (line 517)

所以更准确的说法是：

- 这不是“后面不会影响前面”
- 而是“先把低成本、低破坏面的变量收住，再动高破坏面的变量”
- 每进一层，都要用同一批样本重新评估并重建 baseline，而不是假装旧 baseline 永远有效

文档其实也承认这一点：

- chunk 前必须先有 rebuild / rollback 预案。RETRIEVAL_OPTIMIZATION_BACKLOG.md (line 313)
- 改完要用同一批样本做前后对比。RETRIEVAL_OPTIMIZATION_BACKLOG.md (line 352)
- 如果后续改动导致 retrieval 明显退化，就要停下来恢复证据链，而不是继续往后推。AGENTS.md (line 561)

一句话总结：

这套顺序不是为了“永远控制变量”，而是为了“每一轮只让一个主要因子主导变化”。在工程上，这非常有用；在统计意义上，它当然不是绝对隔离。后面的改动会影响前面，但那正是为什么每进入下一阶段，都必须重新跑同样样本、重建新的 baseline，而不是沿用旧结论。

可以，最核心的因果链是这样：

`Query ↓ 召回范围对不对？ └─ 看 baseline / supplemental 问题：该补不补、该不补乱补 如果这层没稳，后面全是噪音 候选文档对不对？ └─ 仍主要看 supplemental + 候选质量 问题：命中文档是不是正确文档 如果文档都不对，chunk 再漂亮也没用 证据块对不对？ └─ 看 chunk 问题：文档对了，但 top chunk / citation 不准 这时才值得改 chunk 不同 query 类型分流得对不对？ └─ 看 router / hybrid 问题：exact / semantic / mixed 走错支路 这建立在候选和 chunk 已经基本可信之上 最后排序稳不稳？ └─ 看 rerank 问题：候选都不错，但前几名排序不理想 这是最后一刀，不该拿来掩盖上游问题`

所以顺序不是为了说“后面不会影响前面”，而是为了避免这几种误判：

- 召回没稳，就误以为是 chunk 不行
- chunk 没稳，就误以为是 router 分类不行
- 候选本身脏，就误以为 rerank 模型不够强

换成一句工程话就是：

- supplemental 解决“找不找得到、该不该补”
- chunk 解决“找到以后，证据块准不准”
- router 解决“不同问题类型，走哪条候选路径更好”
- rerank 解决“已有候选里谁该排前面”

真正的控制变量方式不是“一劳永逸”，而是：

1. 先稳住一层
2. 固化样本和 baseline
3. 再动下一层
4. 每进一层重新复跑同一批样本
5. 如果退化，就回滚到上一个稳定层

所以它有用的地方在于：

- 能定位主要问题来源
- 能避免把大改动叠在一起后根因不可解释
- 能让每一阶段都有清晰验收面
