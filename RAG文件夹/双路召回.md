
双路召回


始终双路召回，部门结果排前面，其他部门结果排后面” “先查本部门，本部门不够好时才放出其他部门补充”。你觉得我应该做成哪种，从效果和速度考虑

如果只看“上线速度”，选：

- 始终双路召回，部门结果排前面

如果看“实际效果 + 更符合你的业务目标”，我建议选：

- 先查本部门，本部门不足时再补充其他部门

但我会建议你**做一个中间版本**，别一上来做“低分阈值”那种复杂版。

**我建议的最佳方案**  
先做：

- 先查本部门
- 如果本部门结果数量不足，再补充其他部门
- 暂时不要做“分数太低才补充”

也就是把“本部门不够好”先定义成：

- 本部门有效结果数 < top_k
- 或者更稳一点：本部门有效结果数 < rerank_top_n

**为什么我推荐这个**  
从效果看：

- 比“始终双路召回”更符合你的目标
- 更不容易把别的部门内容过早喂给问答模型
- 引用和解释会更自然，用户也更容易理解

从速度看：

- 比“低分阈值补充”简单很多
- 不需要你现在就定分数阈值
- 也不用马上做复杂调参
- 在你当前代码基础上，比完全重写要轻

**三种方案的现实对比**

1. 始终双路召回，部门优先排序
    - 优点：最快，当前代码已经接近这个状态
    - 缺点：别的部门结果会一直混进来，只是排后面
    - 风险：问答模型可能仍然引用跨部门内容，和你的业务目标不完全一致
2. 本部门不足时才补充其他部门
    - 优点：最平衡，效果和开发成本都合适
    - 缺点：要多一步“先看本部门够不够”
    - 结论：我最推荐这个
3. 本部门低分时才补充其他部门
    - 优点：理论上最精细
    - 缺点：最难调，分数阈值很容易拍脑袋
    - 风险：不同 query、不同召回模式下分数口径不稳定
    - 结论：不建议第一版就做

**如果你让我拍板**  
我会这样落地：

1. 第一版做：
    - 先检索本部门
    - 本部门结果数不足时，再补充其他部门
2. 第二版再考虑：

- 当本部门 top1 或 topN 分数过低时，开放补充池

**一句话结论**

- 想快：始终双路召回
- 想更对：先查本部门，不足再补充
- 我建议你现在做第二种，但先只按“数量不足”触发，不按“低分阈值”触发

解释一下supplemental ，supplemental scope ，public_only，same_tenant_retrievable是什么，

可以，我们把这几个词用你这套代码的语境讲清楚。

**1. supplemental 是什么**  
supplemental 就是“补充召回”。

在你的目标里，检索不是一上来全库乱搜，而是分两层：

- 主路：先查本部门
- 补路：如果主路不够，再查别的文档补上

这里第二层补路，就是 supplemental。

所以它不是一种权限角色，也不是一种文档类型，而是：

- “第二优先级的检索来源”

你可以把它理解成：

- primary = 主召回池
- supplemental = 补充召回池

**2. supplemental scope 是什么**  
supplemental scope 就是：

- “哪些文档有资格进入补充池”

也就是补路的边界定义。

比如你现在要决定：

- 补充池里到底放哪些跨部门文档？
- 只放公共文档？
- 还是同租户里所有可检索文档都能进？

这就是在定义 supplemental scope。

一句话：

- supplemental = 补充机制
- supplemental scope = 补充机制允许搜到的范围

**3. public_only 是什么意思**  
public_only 指的是一种补充池策略：

- 只有 visibility == "public" 的跨部门文档，才能进 supplemental pool

也就是说：

- 本部门文档：主路直接查
- 其它部门文档：只有那些被标成 public 的，才允许作为补充

这版的特点是：

- 更安全
- 更保守
- 不容易引发跨部门知识泄露
- 但也更容易漏掉很多本来有价值的企业内部知识

你现在未提交那版 document_service.py (line 1090) 就基本是在做这个：

- department_query_isolation_enabled=True 时
    - 跨部门只有 public 才能进补充池

**4. same_tenant_retrievable 是什么意思**  
这也是一种补充池策略，但范围更大：

- 同一个租户内，只要该文档在“检索场景”下对当前用户是 retrievable 的，就可以进入 supplemental pool

拆开讲：

- same_tenant
    - 只能是同租户
    - 不能跨租户
- retrievable
    - 不是“能直接打开下载”
    - 而是“允许在检索结果里作为候选出现”
    - 它走的是你现在单独拆出来的 retrieval-scoped 权限判断

这意味着在这个策略下：

- 其它部门的文档，不必是 public
- 只要它满足：
    - 同租户
    - role visibility 没拦住
    - retrieval 权限允许
- 就可以在“本部门不足时”拿来补充

这更接近你最开始说的目标：

- 不做部门硬隔离
- 本部门优先
- 其它部门作为补充

**5. 这几个词放到你的业务里怎么理解**  
你现在可以这么记：

- supplemental
    
    - 当本部门结果不够时，用来补位的第二路文档
- supplemental scope
    
    - 哪些文档允许进入这条补位通道
- public_only
    
    - 补位通道里只允许跨部门公共文档
- same_tenant_retrievable
    
    - 补位通道里允许同租户内所有“可检索”的跨部门文档

**6. 两种 scope 的差别**  
如果一个售后部员工在问问题：

本部门命中了 2 条，还不够。

这时：

如果是 public_only

- 只能补充其它部门里标成 public 的文档

如果是 same_tenant_retrievable

- 可以补充其它部门里所有对他“检索可见”的文档
- 不要求这些文档必须是 public

所以：

- public_only 更像“公共知识库补位”
- same_tenant_retrievable 更像“企业内部知识网络补位”

**7. 为什么这事重要**  
因为它直接决定你的企业 RAG 最后像哪一种产品：

如果是 public_only

- 更像“部门优先 + 公共知识库兜底”

如果是 same_tenant_retrievable

- 更像“部门优先 + 企业内部跨部门知识协同”

后者更强，但也更需要你确认合规边界和组织接受度。

- 用户是谁
- 文档在哪个部门
- 文档 visibility 是什么
- 在哪种策略下能不能进 supplemental pool

**第三层：按部门定向授权检索**

- 最适合企业
- 可以规定“这个部门的文档允许被哪些部门检索”
- 是你刚刚说的这个方向

如果我从产品和架构角度给建议：

- 短期可以先从 same_tenant_retrievable 起步，先把“本部门优先、不足补充”跑通
- 中期最好升级成“定向部门可检索”模型

因为 same_tenant_retrievable 很适合验证价值，但长期会有两个问题：

- 面太大
- 很难解释“为什么这个部门能检索到那个部门的知识”

而“定向部门可检索”就能解释得很清楚：

- 文档归属：生产技术部
- 可直接访问：生产技术部
- 可检索部门：质量部、售后部、设备部

这个设计是企业里最容易落地的。

一句话总结：

- 对，same_tenant_retrievable 比 public_only 更全面
- 但你说的“可以规定自己部门信息被哪些部门可检索”，比 same_tenant_retrievable 还更好，也更企业级

- 该加什么字段
- direct access 和 retrieval access 怎么拆
- 对当前代码最小改哪些地方
