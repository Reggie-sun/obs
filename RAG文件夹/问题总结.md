
总结一下为什么上次rebuild会出现这个索引丢失的情况

3:09 PM

上次 rebuild 会出现“索引丢失”，本质上不是单一原因，而是两类问题叠在一起了：

1. rebuild 本身是“先删旧点，再重新入库”
    - POST /documents/{doc_id}/rebuild 会先按 document_id 删除 Qdrant 旧点，再创建 follow-up ingest job。
    - 所以只要后续入库没走完，这个文档就会从“原本可检索”瞬间变成“Qdrant 0 点”。
2. 后续入库失败时，旧代码有两种失配会把 metadata 和索引状态撕开

第一种：第三方异常没被收口

- 典型就是 Qdrant client timeout。
- 旧的 run_ingest_job() 只 catch ValueError / RuntimeError / OSError。
- 但 Qdrant 超时抛的是库自己的 ResponseHandlingException，不在 catch 范围内。
- 结果就是：
    - worker 任务直接炸掉
    - job metadata 停在 indexing 90%
    - 文档状态也没被正确打成 failed/dead_letter
    - 而旧点早就被删掉了，所以看起来就是“索引丢失”。

第二种：embedding 输入过长，外部 embedding API 直接 400

- 有些结构化 chunk，尤其 clause / OCR 合成文本，会非常长。
- 之前没有对单条 embedding 输入做长度保护。
- DashScope 那边对过长输入直接 400 Bad Request。
- 于是 job 会进 dead_letter，而因为 rebuild 开头已经删了旧点，所以文档也会变成 0 点。

再往下还有一个“放大器”：

3. metadata / Qdrant 之前缺少一致性防线
    - 旧逻辑里，即使 job 已经 failed/uploaded/indexing，检索 scope 也没有严格排除这些非 ready 文档。
    - 另外，如果服务端已经写入了部分点，但 client 侧超时，metadata 可能是 failed，Qdrant 却还残留点。
    - 这就会出现两种相反的脏状态：
        - 文档是 live，但 Qdrant 没点
        - 文档不是 live，但 Qdrant 还有点

所以你看到的“索引丢失”，准确说是这条链路：

- rebuild 先删旧点
- 新入库过程中因为 Qdrant timeout 或 embedding 400 失败
- 失败没有被完整收口，或者只收口了 metadata 没收口点位
- 最后变成 metadata 和实际索引不一致

这次我们修掉的是这几件事：

- 第三方异常统一收口到 failed/dead_letter
- 超长 embedding 输入先截断再送模型
- 失败时主动清理该文档残留点
- 检索 scope 只允许 active / partial_failed 文档参与召回
- 最后又把 live corpus 做了一轮补 rebuild 和残留清理

所以现在这个“rebuild 后索引丢失”的原始问题，已经不是 active blocker 了。

可以，这就是我建议的安全操作法。

按这个顺序来：

1. 第一波：txt / md / docx
    - 每批 10-20 个
    - 先跑最轻文档，确认 worker、embedding、Qdrant 都稳定
2. 第二波：xlsx / pptx / html / json / csv
    - 还是每批 10-20 个
    - 这类通常比 OCR 文档稳，但结构化解析偶尔会有边角问题
3. 第三波：pdf
    - 先分“原生文本 PDF”和“疑似扫描 PDF”
    - 原生文本 PDF 可以一批 10 左右
    - 扫描/OCR 重文档建议一批 5-10
4. 第四波：图片类 OCR 文档
    - png / jpg / jpeg / webp / bmp
    - 这一波最慢，建议 5 个左右一批

建议执行方式：

`python scripts/rebuild_documents.py \ --api-base http://127.0.0.1:8020 \ --document-list /tmp/rebuild_batch_01.txt \ --status-file /tmp/rebuild_batch_01_status.json`

然后等这批结束：

`python scripts/rebuild_documents.py \ --api-base http://127.0.0.1:8020 \ --status-file /tmp/rebuild_batch_01_status.json \ --wait --poll-interval 10 --max-poll-attempts 120`

每一波结束后看这几件事：

- completed / partial_failed / dead_letter 各有多少
- 有没有新出现的 live_zero_docs
- 有没有 non_live_has_points
- worker 日志里有没有连续 timeout / 400 / Qdrant 异常

我的建议批次大小：

- txt/md/docx: 15-20
- xlsx/pptx/html/json/csv: 10-15
- 原生文本 pdf: 10
- OCR 重 pdf / 图片: 5-8
