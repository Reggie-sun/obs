这个文件实现了 **文档管理服务**，是企业级 RAG 系统的核心模块之一。它负责：

1. **文档全生命周期管理**：上传、创建、预览、下载、删除、向量重建
2. **异步入库任务调度**：通过 Celery 谬排后台入库任务
3. **部门数据隔离**：多租户场景下的部门级权限控制
4. **RBAC 权限控制**：基于角色的访问控制
5. **元数据存储**：支持 JSON 文件和 PostgreSQL 两种后端

---

### 🔧 常量与数据类

|名称|作用|
|---|---|
|`SUPPORTED_UPLOAD_SUFFIXES`|支持上传的文件扩展名（与可解析扩展名一致）|
|`TEXT_PREVIEW_SUFFIXES` / `PDF_PREVIEW_SUFFIXES`|支持预览的文件类型|
|`PROCESSING_JOB_STATUSES`|入库任务进行中状态：`parsing`, `ocr_processing`, `chunking`, `embedding`, `indexing`|
|`IN_FLIGHT_JOB_STATUSES`|进行中状态（含待处理）|
|`TERMINAL_JOB_STATUSES`|终态：`completed`, `dead_letter`, `partial_failed`|
|`DepartmentPriorityRetrievalScope`|**部门优先检索作用域**，包含 `department_document_ids`（本部门）和 `global_document_ids`（补充池）|
|`DocumentFilePayload`|文件下载/预览的返回结构（路径或字节内容）|

### 🏗️ `DocumentService` 类核心方法

#### 1. 文档创建与上传

|方法|功能|
|---|---|
|`create_document()`|**主入口**：处理文件上传、重复检测、权限校验、任务调度|
|`create_documents_batch()`|批量上传，逐个调用 `create_document`|
|`save_upload()`|旧版同步上传入口（直接入库，不创建元数据记录）|

**`create_document()` 核心流程：**

```
1. 读取上传内容 → 校验文件类型和大小
2. 标准化参数（tenant_id, department_id, visibility 等）
3. 权限校验 → _enforce_document_create_scope()
4. 重复文件检测 → 通过 file_hash 判断同租户是否已有相同内容
   ├─ 有重复 → 智能复用已有文档/任务
   └─ 无重复 → 创建新的 DocumentRecord + IngestJobRecord
5. 保存元数据 → PostgreSQL 或 JSON 文件
6. 调度入库任务 → dispatch_ingest_job()
7. 记录事件日志 → _record_document_event()
```

### 🔐 权限控制方法

|方法|作用|
|---|---|
|`_can_read_document()`|**直接访问权限**：详情/预览/下载，检查租户边界 + 部门隔离 + role visibility|
|`_can_retrieve_document()`|**检索权限**：只检查租户边界 + role visibility，**不做部门隔离**，让跨部门文档能进入补充池|
|`_can_manage_document()`|**管理权限**：仅 `sys_admin` 或文档所在部门的 `department_admin` 可操作|
|`_ensure_can_read_document()`|权限断言，不通过则抛 403|
|`_ensure_can_manage_document()`|管理权限断言|

**权限检查逻辑：**

```
_can_read_document():
  1. auth_context 为空 → 允许（开发/调试场景）
  2. 租户不匹配 → 拒绝
  3. role.data_scope == "global" → 允许（系统管理员）
  4. visibility == "role" 且用户角色不在 role_ids → 拒绝
  5. visibility == "public" → 允许
  6. department_query_isolation_enabled == False → 允许
  7. 文档部门与用户可访问部门有交集 → 允许

_can_retrieve_document():
  1-4 同上
  5. 直接返回 True（允许跨部门检索）
```

### 🎯 部门优先检索作用域

```python
def build_department_priority_retrieval_scope(self, auth_context) -> DepartmentPriorityRetrievalScope:
```

**功能**：为非全局角色用户构建检索作用域，供 `retrieval_service` 实现分阶段召回。

**返回结构**：

- `department_document_ids`：本部门可检索文档 ID 列表
- `global_document_ids`：**补充池**（跨部门公开文档）

**补充池条件**：

- 当 `department_query_isolation_enabled=False` 时：所有同租户可检索文档
- 当 `department_query_isolation_enabled=True` 时：仅 `visibility="public"` 的跨部门文档

### 📊 状态机

```
[queued] → [uploaded] → [parsing] → [ocr_processing] → [chunking] → [embedding] → [indexing] → [completed]
                ↓
            [failed] → (重试) → [dead_letter]
                ↓
            [partial_failed]  (OCR 有告警但有可用结果)
```

---

### 🔄 重复上传处理逻辑

1. **完全相同内容已存在** → 计算文件哈希，匹配同租户已有文档
2. **已有任务正在进行** → 复用 in-flight 任务（避免重复入库）
3. **已有任务完成且有产物** → 直接返回 `status="completed"`
4. **已有任务完成但无产物** → 创建 follow-up 任务重新入库
5. **已有任务卡住但有产物** → 自动修复为 `completed` 状态

### 🗃️ 元数据存储

- **PostgreSQL 模式**（`postgres_metadata_enabled=True`）：
    
    - 使用 `PostgresMetadataStore` 进行 CRUD
    - 支持批量查询（避免 N+1）
- **JSON 文件模式**（默认）：
    
    - 每个文档/任务一个 JSON 文件
    - 路径：`{document_dir}/{doc_id}.json`、`{job_dir}/{job_id}.json`

### 📝 事件日志

所有文档操作都会调用 `_record_document_event()` 记录事件：

- `action`: `create`, `batch_create`, `delete`, `rebuild`
- `outcome`: `success`, `failed`
- `duration_ms`: 操作耗时
- `details`: 额外元信息（文件名、是否重复、向量点删除数量等）
