

现在删除都是只删除索引，我怎么重新入库

Worked for 1m 4s

不是“只删索引”。现在这套实现其实是：

- DELETE /api/v1/documents/{doc_id} 会把文档状态改成 deleted，同时删掉向量点位，见 document_service.py (line 905)。
- 但**原始源文件并没有被物理删掉**，后续入库任务还是会从 storage_path 物化源文件再走一遍解析/切块/向量化，见 document_service.py (line 614)。

所以你重新入库分两种情况：

1. 文档还没删，只是想用新 cleaning / 新 chunk 配置重建  
    直接用 rebuild，不要先删。
    - 单个文档：POST /api/v1/documents/{doc_id}/rebuild
    - 批量自己的文档：POST /api/v1/documents/rebuild
    - 脚本入口：scripts/rebuild_documents.py
2. 文档已经被删成 deleted  
    这时**不能 rebuild**，代码会直接返回 409，见 document_service.py (line 1125) 和相关测试 test_document_ingestion.py (line 1173)。  
    你要先走 restore：

- POST /api/v1/documents/restore

restore 会把你当前用户名下、状态为 deleted 的文档改回 uploaded，然后创建 follow-up ingest job 重新入库，见 document_service.py (line 1079) 和 documents.py (line 230)。

你现在做 cleaning canary，我建议操作是：

- 不要删文档
- 开启 cleaning 开关
- 对 canary 文档直接跑 rebuild

如果你已经删了，那就：

- 先 restore
- 再看 job 状态
- 必要时再对恢复后的活跃文档做 rebuild

最实用的一句话是：

**想重新入库，优先用 rebuild；已经删成 deleted 的，先 restore，不要靠 delete 之后再 rebuild。**

这套逻辑有必要改吗

短答：**核心逻辑现在不一定要改，我更倾向于先不改。**

原因很直接：

- 这套生命周期其实已经能支持你现在的 Phase 2B 需求：
    - 活跃文档用 rebuild
    - 已删文档用 restore
- 对 cleaning canary 来说，这种“保留源文件 + 软删除 metadata/status + 清理向量”的设计反而是有利的，因为你可以重复入库，不需要重新上传原文件。
- 现在去改这套逻辑，会碰文档生命周期、权限、任务调度、测试面，风险比收益大。

我觉得**不该现在改的**是：

- delete = 软删除 + 删索引
- rebuild = 活跃文档重入库
- restore = 已删文档恢复并重入库

这三层语义本身是成立的。

但我觉得**有一处值得以后优化**：

- 现在“操作体验”有点绕，不是“底层逻辑错了”。
- 特别是 restore 目前是批量恢复“我删除的文档”，而不是按 doc_id 精准恢复，这在 canary 或运维场景里不够顺手。
- 所以后续如果要改，我会优先改：
    - 文档/命令说明
    - 一个更明确的单文档 restore/reingest 入口
    - 或 CLI 包装
- 而不是重写 delete/rebuild 的生命周期语义。

我的判断是：

- **现在没必要改核心逻辑**
- **如果你觉得难用，优先改操作入口和可用性，不要先改生命周期模型**

一句话版：  
**问题更像“入口不顺手”，不是“底层逻辑必须重做”。**
