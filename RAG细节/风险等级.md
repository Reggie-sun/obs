
P0

- **拆分 retrieval 核心策略层**。backend/app/services/retrieval_service.py 已经 5048 行，而仓库自己的计划也明确把它视为“高复杂度区”。现在继续往里叠 ranking/promotion/fusion 规则，收益会越来越小，风险会越来越高。
- **把 eval 的“可信度前提”做成硬门槛**。eval/README.md、RETRIEVAL_OPTIMIZATION_BACKLOG.md 已经反复提到 ACL seed、active snapshot、governance、metadata store 对齐问题。你现在最怕的不是分数低，而是 37/37 看起来漂亮，但其实不是同一个 runtime truth。
- **把 citation grounding 当成产品级正确性问题处理**。尤其是 fast + memory 这条线，答案正文“像对了”但 citation 错，是企业场景里最伤信任的一类问题。这件事我会放在 retrieval 继续调参之前。

P1

- **拆 ingestion/chunk 链路的第二批大文件**。backend/app/services/ingestion_service.py 3224 行，backend/app/rag/chunkers/structured_chunker.py 1373 行。你们现在很多“检索问题”其实迟早会回流到 ingestion 表达层。
- **补几条真正的黄金端到端链路**。不是再加一堆 unit test，而是固定少数高价值流：上传 -> 入库 -> 预览 -> chat -> citation -> portal/workspace 可见性。这件事对“看起来改了但用户没感知”“引用对不上”“权限态漂移”很有用。
- **收紧 control-plane / runtime 复杂度**。这个仓库的规则、lane、branch runtime、snapshot/governance 已经很强，但也开始偏重。下一步要避免“系统为了保证正确而变得只有作者自己能驾驭”。

P2

- **整理 document/chat citation 边界**。backend/app/services/document_service.py 和 backend/app/services/chat_citation_pipeline.py 这类文件后面很容易出现职责缠绕。
- **统一 portal/workspace 的状态模型**。现在两条前端线都在长，如果没有持续抽 view-model / pure model seam，后面维护成本会慢慢抬升。
- **暂缓继续追 retrieval 分数型优化**。在 P0 没补之前，再追更多 case、更多局部 heuristics，我觉得投入产出会变差。
