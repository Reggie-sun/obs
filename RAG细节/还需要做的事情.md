2回答失败半自动化


> **代码自动收集证据 + 你人工判断失败类型 + 后面逐步半自动化。**

不要幻想一开始系统自己知道“为什么答错”。系统只能记录现象，真正判断原因通常要人看。

## 正确做法是这样

代码可以自动记录这些：

```
用户问题rewrite 后的问题命中的 top chunks命中文档rerank 分数最终引用LLM 回答用户点赞/点踩trace_id耗时权限过滤结果
```

这些是**客观日志**，可以自动生成。

但是下面这些，通常需要你人工标注：

```
标准答案是什么应该命中哪个文档/条款实际为什么错是解析问题，还是检索问题，还是 rewrite 问题是否文档本身缺失是否用户问题超范围是否 LLM 胡编
```

这部分不能完全交给代码。

## 你应该怎么做

第一阶段，不要搞复杂系统。  
直接用 Excel / 飞书表格 / YAML / CSV 都行。

字段可以这样：

```

```

其中最重要的是这几个：

```
queryexpected_docactual_topk_docsis_correctfailure_typefix_action
```

够用了。

## failure_type 先手工标

比如你看一个失败问题：

用户问：

> “A 设备报警 E03 怎么处理？”

系统答不上来。

你看 trace：

- 知识库里没有 E03 文档；
- 那就是：`文档缺失`

如果文档里有，但 OCR 没识别出来：

- 那就是：`解析/OCR失败`

如果 OCR 有，chunk 里也有，但没召回：

- 那就是：`检索失败`

如果召回了，但排在第 9，rerank 没排上来：

- 那就是：`rerank失败`

如果召回和引用都对，但回答乱说：

- 那就是：`生成失败`

这必须你人工看。

## 后面可以半自动化

等你积累 50～100 条失败样本后，可以让代码帮你自动生成初步判断。

比如规则：

```
expected_doc 不在 topk 里 → retrieval_failureexpected_doc 在 topk 里但不在 final_context → rerank/context_failureexpected_doc 在 final_context 里但回答错 → generation_failureexpected_doc 为空 → doc_missingOCR 文本为空或乱码比例高 → parse_ocr_failurerewrite 后 query 与原 query 偏离太大 → rewrite_failure
```

但注意，这只是**辅助判断**，不是最终真相。
