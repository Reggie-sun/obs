如果你问的是“**拿哪一类样本去驱动清洗/检索优化**”，这个仓库里主要看这几个路径，作用不一样：

**1. 主 truth 样本**

- eval/retrieval_samples.yaml

这是最核心的样本库。  
这里的样本定义了：

- query 是什么
- 期望命中哪个 doc_id
- 期望 chunk_type
- 是 frozen 还是 rolling

用途：

- 跑 scripts/eval_retrieval.py
- 做回归验证
- 判断某次清洗/检索改动是不是把结果带偏了

如果你要优化清洗或检索，**最终都要回到这个文件里的样本来验收**。

**2. 优化 case 入口**

- eval/optimization_cases.yaml

这是“要不要开优化 lane”的入口，不是最终 truth。  
它更像 case 管理表，告诉你：

- 当前问题属于 cleaning、index_governance 还是别的层
- 该不该先补治理证据
- 该不该进实现优化

用途：

- 跑 scripts/eval_optimization_cases.py
- 做 triage
- 决定是不是值得动 cleaning/retrieval 代码

如果你现在要“找一个样本来优化清洗”，**先看这里有没有对应 case**。

**3. 当前窗口这批问答样本的草案/预览**

- eval/generated/parallel_question_investigation_verified_sample_draft_20260512.yaml
- eval/generated/parallel_question_investigation_promotion_preview_20260512.yaml

这些是这轮收口过程中生成的**草案或预览**。  
它们可以帮助理解问题，但**不能当成优化清洗的正式 truth 主入口**。

有意义，而且意义很大。你的 eval/ 里不应该只有“最终验收测试集”，否则你会很容易把优化做成两种坏情况：

- 只会刷分 frozen/verified，但对真实用户 query 反应迟钝
- 把 runtime/corpus drift、governance 问题、snapshot 不可比，误判成 retrieval 算法退化

可以把它们理解成 4 层。

**1. Gate Truth**  
这是最后用来证明“没带偏系统”的。  
典型就是 eval/retrieval_samples.yaml 里 track=frozen、status=verified 那部分，以及对应 eval/results/* 的 frozen gate 结果。

它的意义是：

- 做最终通过/不过的判断
- 防止你为了修一个真实 query，把主线能力打坏
- 给每次优化提供统一基线

**2. Live Observation**  
这是“真实世界雷达”，不是最终裁判。  
像 rolling、candidate、真实库 snapshot 派生出来的 seed，都属于这层。

它的意义是：

- 捕捉真实用户新问题
- 看当前 active corpus 下哪些文档家族开始漂移
- 发现 frozen gate 没覆盖到的新 query surface

没有这层，你只能优化旧题，越来越脱离真实使用。

**3. Triage / Diagnosis**  
这层是最容易被忽视，但其实最值钱的。  
像 eval/optimization_cases.yaml、trace_candidates、各类 decision preview / bundle，都是为了回答：

- 这是 cleaning 问题吗？
- 是 chunk 问题吗？
- 是 retrieval 问题吗？
- 还是根本是 active snapshot / governance / family drift 问题？

它的意义是：

- 不让你看到一个差结果就直接改代码
- 把“现象”变成“有证据的单目标 case”
- 降低误修概率

**4. Comparability / Governance**  
这是防止“假回归”的护栏。  
像：

- eval/library_document_snapshot.yaml
- eval/rolling_corpus_governance.yaml
- alignment report
- family truth registry

它们的意义是：

- 先确认当前 runtime 里是不是还是同一批文档
- 确认新 top1 是不是同 family 漂移
- 确认某个 query 失败是不是因为 expected doc 已经不在 active corpus

没有这层，你会经常把“数据变了”错当成“排序坏了”。

所以一句话总结：

**最终测试集负责“放行”，其它 eval 文件负责“发现问题、解释问题、定位问题、决定该不该修”。**

如果只保留 frozen 测试集，你会得到一个很严谨但很迟钝的系统。  
如果只看 rolling / live query，你会得到一个很灵敏但很容易乱改的系统。  
现在这套 eval/ 的价值，就是把这两者隔开：

- frozen/verified 决定能不能过
- rolling/candidate/query-driven cases 决定下一步该看哪里
- snapshot/governance/alignment 决定这个问题值不值得进实现面

你现在最好的用法就是：

1. 真实用户 query 先进 sample-optimization 或 library-sample-intake
2. 先走 compact summary
3. 如果只是新现象，先落到 case / preview / governance
4. 只有证据够了，才进入实现
5. 最后再用 frozen gate 验证没回退
