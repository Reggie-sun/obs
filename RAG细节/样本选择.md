
为什么需要这样严谨的选择样本问题

10:34 AM

因为这一步不是“随便找几份文档试试”，而是在决定我们后面看到的结果，到底是在测系统能力，还是在测样本噪音。

最核心的原因有几个。

第一，样本会直接决定你看到的是“真进步”还是“假进步”。  
如果把 rolling 和 frozen 混在一起，我们很容易一边拿样本调系统，一边再用同一批样本宣布系统变好了。那种提升没有说服力，只是“做过题再考试”。

第二，这个仓库当前有真实的 runtime/corpus 漂移历史。  
前面已经发生过 live runtime 和 frozen baseline 不可比的情况，所以现在必须先确认：

- 跑的是哪套实例
- 文档到底在不在当前库里
- 命中的是不是同一个 family  
    不然你看到一次命中失败，可能根本不是 retrieval 变差，而是样本对应的文档不在这套 runtime 里了。

第三，合同类问题非常容易被“表面答对”骗过去。  
比如用户问“合同金额是多少”，系统可能确实返回了一个金额，但那个金额可能来自：

- 标题附近的普通文本
- 错误的表格行
- 另一个相似文档
- 泛化召回出来的 global text  
    这和真正命中 clause 或 table 不是一回事。我们现在要验证的是 contract_like 结构化能力，不只是“看起来像答对了”。

第四，错误样本会把优化方向带偏。  
比如采购文件、报价附件、招标文档，当前治理里本来就被当成 procurement_artifact 或 commercial_attachment 排除。如果把这类文档硬塞进主样本池，后面你为了救这些样本去调规则，最后优化的可能不是合同检索，而是采购附件检索，方向就偏了。

第五，这一步是在保护后面的工程成本。  
样本没选严，后面每次失败你都不知道该怪谁：

- 是文档本身不适合
- 是 truth 没锁定
- 是 runtime 不对
- 是 chunk 没结构化
- 还是 retrieval 真退化了  
    一旦这个边界不清，后面每次调试都会非常贵。

所以严格选样本，本质上是在回答一个问题：  
“我们后面测到的好坏，能不能被相信？”

在这个仓库里，严谨选样本不是流程洁癖，而是为了避免把：

- runtime 漂移
- family 混池
- procurement artifact
- 假阳性命中  
    误判成系统能力变化。

不行，关键不是“是不是人挑的”，而是“这批样本有没有参与过优化决策”。

如果你人为挑样本，然后拿这批样本来决定：

- 哪类 family 值得留
- 哪类 query 该怎么写
- 哪个文档先升 rolling
- 哪种命中算可接受
- 哪些失败先忽略

那这批样本就已经被“看过、用过、影响过系统选择”了。它就不再是 frozen，哪怕你一行参数都还没改。

最简单地说：

- rolling/canary 是“拿来帮我们做判断和调优的样本”
- frozen 是“等一切决定做完后，拿来验收的样本”

区别不在于是不是人工选，而在于有没有被用于优化过程。

拿我们现在这条线举例：

- 我们已经因为 live probe 结果，调整了 DFMS 和 拓璞 的优先级判断
- 已经决定 procurement artifact 不能混进来
- 已经决定 payment 暂时不绑定
- 已经根据命中 locus 去判断谁更适合首轮 canary

只要这些判断是基于某批样本做出来的，这批样本就已经“沾了优化过程”，不能再直接升 frozen。

否则会出现一个假象：  
你以为是在做正式评测，其实是在测“我们刚刚围着它做过决策的那批样本”。

更直白一点：

- frozen 不是“人没碰过”
- frozen 是“人没有拿它来指导本轮优化”

所以正确顺序还是：

1. 先人工挑一批 candidate
2. 从里面选 rolling/canary，用来做 intake、probe、调优判断
3. 另外单独留一批不参与这些判断的 family
4. 等规则、样本边界、执行方式稳定后，再把那批未参与优化的样本定成 frozen

一句话总结：

你可以人工选 frozen，但不能把“已经参与过我们优化思考的样本”直接叫 frozen。

- live 里是不是这份文档
- 它的 chunk 到底是什么形态

因为现在 DFMS 最大的不确定性已经不是“问什么”，而是“它到底被怎么存进去”。

如果 chunk 形态不对，你写再多 query 都是在测一个错的前提。

具体说：

用户问“合同金额是多少”，我们理想里希望发生的是：

1. 文档先被 contract_like 结构化入库
2. 里面有清晰的 clause 或 table
3. retrieval 看到这是金额问法
4. branch policy 把候选往 clause/table 方向推
5. top1 命中真正的金额条款

但 DFMS 现在 live 上实际发生的是：

- 文档在库里
- 问金额时 top1 还是 text
- clause_branch_candidates=0
- table_branch_candidates=0

这说明问题很可能出在前面几步，而不是“query 不够聪明”。

换句话说，现在更像是：

- 不是“问法没设计好”
- 而是“系统压根没看到它是一个可用的 clause/table 文档”

所以这时继续写 query 没什么价值，因为你只会不断重复得到：  
“它还是命中 text。”

这不能帮我们区分到底是哪种问题：

- 这份 live 文档根本没走 contract_like
- 它走了，但 chunk 没正确落出 clause/table
- chunk 落了，但索引里没反映出来
- 索引有了，但 retrieval 取候选时没拿到
- 还是 branch policy 没真正吃到这些候选

而查 chunk 形态就能直接把问题缩小。

如果一查发现：

- 根本没有 clause/table  
    那就不是 query 问题，是 ingest/chunk 问题
- 有 clause/table，但检索拿不到  
    那就是索引/候选召回问题
- 能拿到 clause/table，但 top1 仍是 text  
    那才轮到 branch policy / ranking 问题

这就是为什么“先查 chunk 形态”比“继续补 query”更值钱。

它不是更保守，而是更省时间。  
因为它能一下子告诉我们：后面应该去查哪一层。

一句最实操的话就是：

现在 DFMS 不是缺题目，而是缺“确认它有没有被正确做成这道题”。

所以下一步先查 chunk 形态，本质上是在先确认：  
“这份 live 文档到底有没有资格参加我们这轮 contract_like canary。”

**名词解释**

- rolling: 这里就是“当前库里这批文档的观察集”。它不是正式 gate，但用来盯现在这批语料是否还能被搜到。
- candidate: 就是“先观察、先别并入主观察集”的候选区。之前有一些文档放这里，现在已经清空了。
- family_merge: 指“同一份内容有多个 active 版本”，比如 pdf/pptx。这类不算检索退化，而是按 family truth 接受。
- exclude: 指“这份文档虽然在库里，但不该拿来定义检索健康度”，常见是报价、历史稿、参考资料、招标文件之类。
- curated query: 不是随便从标题里自动长出来的 query，而是根据文档正文特征，人工挑出来、并正式写进治理规则的 query。它这次很关键，因为 01303 这份文档默认 query 不稳，但 curated query 稳。
- alignment report: 就是检查 snapshot / governance / retrieval_samples 三者有没有打架。这次结果是对齐的。
- checked-in diff: 就是“如果现在按规则重新生成样本，会不会和仓库里已经提交的样本文件不一致”。这次是 clean。
